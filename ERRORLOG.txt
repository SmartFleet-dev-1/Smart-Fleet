SMART FLEET PRO — PROJECT ERROR TRACKING LOG

Purpose:
This log tracks the major errors faced during Smart Fleet Pro development and how each issue was identified and fixed. This can be used for interview explanation, daily tracking, project documentation, and RustPad notes.

------------------------------------------------------------
DAY 1 — BACKEND REPOSITORY ERRORS
------------------------------------------------------------

Error Faced:
Maven test failed with errors like:

cannot find symbol method deleteAll()
cannot find symbol method save()
cannot find symbol method findAll()
cannot find symbol method findById()

The errors appeared for repositories such as:
VehicleRepository
DriverRepository
InsuranceRepository
MaintenanceRecordRepository
UserRepository

Root Cause:
Repository interfaces were not properly extending JpaRepository.
Because of that, Spring Data JPA default methods were unavailable.

Fix Applied:
Updated all repository interfaces to extend JpaRepository<Entity, Long>.

Example:
public interface VehicleRepository extends JpaRepository<Vehicle, Long> {
}

Result:
Repository-level compile errors were resolved.

Interview Explanation:
Initially, repository methods like save, findAll, findById, and deleteAll were unavailable because the repository interfaces were not extending JpaRepository properly. After updating all repositories to extend JpaRepository, Spring Data JPA provided those methods automatically.

DAY 5 — USER ROLE TYPE ISSUE
------------------------------------------------------------

Error Faced:
UserService initially treated role as String, but User entity used enum Role.

Wrong operations included:
role.trim()
role.startsWith()
user.setRole("ADMIN")

Root Cause:
Role was an enum, not a String.

Fix Applied:
Updated role handling:

Role role = user.getRole();

if (role == null) {
    role = Role.ADMIN;
}

String roleName = role.name();

Used:
user.setRole(Role.ADMIN);

Instead of:
user.setRole("ADMIN");

Result:
User authentication and Spring Security authority generation worked.

Interview Explanation:
The role field was mapped as an enum using @Enumerated(EnumType.STRING). I updated the service to use enum-safe logic and converted the role to string only when creating Spring Security authorities.

DAY 11 — REGISTRATION FAILED WITH GENERIC ERROR
------------------------------------------------------------

Error Faced:
Frontend showed:

Registration failed. Username/email/contact may already exist.

Root Cause:
Frontend displayed a generic message for all registration failures. MySQL showed the users table was empty, so duplicate data was not the issue.

Diagnosis:
Browser Network tab showed:

POST /api/auth/register → 403 Forbidden

Response preview showed:

Invalid CORS request

Fix Applied:
Updated Spring Security to allow auth APIs and OPTIONS requests:

.antMatchers(HttpMethod.OPTIONS, "/**").permitAll()
.antMatchers("/api/auth/**").permitAll()

Updated global CORS config:

.allowedOriginPatterns("*")
.allowedMethods("GET", "POST", "PUT", "DELETE", "PATCH", "OPTIONS")
.allowedHeaders("*")
.exposedHeaders("*")
.allowCredentials(false)

Result:
Registration started working successfully.

Interview Explanation:
The registration failure was caused by CORS and Spring Security, not duplicate data. The frontend and backend were running on different proxy ports, so the browser treated them as different origins. I fixed this by allowing CORS globally and permitting auth endpoints in Spring Security.

DAY 13 — LOGIN FAILED WITH JWT ERROR
------------------------------------------------------------

Error Faced:
Login API returned:

{
  "message": "Illegal base64 character: '_'",
  "status": 500
}

Root Cause:
JWT secret key contained an underscore:

smartfleetpro_secret_key_2026

With JJWT 0.11.2, the signing logic caused a Base64-related error.

Fix Applied:
Updated JwtUtil.java to use a proper HMAC signing key:

private final String SECRET_KEY = "smartfleetprosecretkey2026securejwtkey";

private Key getSigningKey() {
    return Keys.hmacShaKeyFor(SECRET_KEY.getBytes(StandardCharsets.UTF_8));
}

Updated JWT signing:

.signWith(getSigningKey(), SignatureAlgorithm.HS256)

Result:
Login started working and JWT token was generated successfully.

Interview Explanation:
The JWT secret format caused a signing error. I fixed it by generating a proper HMAC signing key using Keys.hmacShaKeyFor() and a secure string converted to UTF-8 bytes.

After successful login, I copied the JWT from localStorage and verified it on jwt.io using the backend signing secret. This confirmed that the token was valid, signed using HS256, and contained the expected username subject and expiry claims.

INTERVIEW SUMMARY
------------------------------------------------------------

During integration, multiple issues occurred across backend, 
frontend, database, security, and JWT. I debugged them step-by-step using Maven logs, 
MySQL queries, browser Network tab, backend logs, and jwt.io.

and JWT signing failure due to an invalid secret format. After fixing these issues, 
the complete registration-login-dashboard flow worked successfully with MySQL storage and 
verified JWT authentication.
